# 统一运行时上下文与自动返回值提取设计规范

**版本**: v3.0  
**日期**: 2026-04-07  
**状态**: 设计完整版（草案）

---

## 0. 本设计要解决的问题

RodSki 当前的运行时取值与步骤返回值机制存在以下设计问题：



1. **缺少更直接的命名存取方式**  
   业务流程中经常会产生多个重要中间值，例如两个订单号、两个 token、多个业务单据编号。仅靠 `${Return[-1]}` / `${Return[-2]}` 难以稳定表达“我要的是哪一个值”。

2. **当前 Return 的写入覆盖不完整且类型不统一**  
   部分关键字执行后没有写入 Return，导致步骤链断裂；同时 `evaluate` 等关键字对结果做了不一致的字符串化处理，破坏了后续引用的一致性。

3. **业务关键返回值获取过于依赖显式步骤**  
   在 UI 自动化场景中，很多业务步骤执行后会立即产生关键业务结果，例如“创建询价单后生成询价单号”“下单后生成订单号”。当前用户往往需要额外再写 `get` 步骤去获取这些值，导致用例啰嗦，并且把框架本应承担的模型能力转嫁给用户。

4. **`evaluate` 关键字存在滥用风险**  
   `evaluate` 能力虽然对 Web 场景有实用价值，但如果没有清晰约束，容易成为绕开模型驱动与 DSL 语义的逃生舱，最终侵蚀框架的一致性与可维护性。

5. **需要一个统一的运行时数据模型**  
   `Return`、`set`、`get` 不应被理解为三套机制，而应统一到一个运行时上下文中：步骤结果、命名变量、未来的结构化对象，都应归属于同一个上下文，只是访问方式不同。

**因此，本设计的目标不是再引入一套独立“数据空间”，而是定义一个统一的运行时上下文（Runtime Context）模型，并在此基础上引入模型驱动的自动返回值提取能力。**

---

## 1. 设计目标

- 定义 **统一运行时上下文**，作为测试用例执行期间所有运行时值的唯一归属空间
- 保留 `${Return[-N]}` 的相对引用能力，兼容“步骤无编号”的原始设计目标
- 明确 `set` / `get` 是对统一上下文的命名访问能力，而不是独立于 Return 的另一套系统
- 统一所有关键字的返回值写入规则，确保每个测试步骤执行后都能形成可追溯的运行时记录
- 引入**模型驱动的自动返回值提取机制**，在关键业务步骤完成后自动获取关键值并写入运行时上下文
- 将 `get` 定位为**显式补充读取能力**，覆盖自动提取未覆盖到的场景
- 将 `evaluate` 定位为 **Web-only 的受限兜底能力**，并明确其使用约束，避免滥用
- 为未来结构化路径访问预留扩展能力，但首版不引入独立的数据空间机制

---

## 2. 核心设计结论

### 2.1 只有一个运行时上下文

RodSki 在单个 case 执行期间维护一个统一的 **Runtime Context**。

这个上下文中包含运行期间产生的所有值，包括：
- 每一步关键字执行后的结果
- 用户通过 `set` 保存的命名变量
- 自动返回值提取得到的关键业务值
- 未来可能扩展的结构化对象值

**结论**：
- `Return`、`set`、`get` 都不是独立存储空间
- 它们只是对同一个运行时上下文的不同访问方式
- 从本质上看，`${Return[-1]}` 和 `get` 都是“取值操作”，只是访问的逻辑视图不同

---

### 2.2 Return 是统一上下文的“历史视图”

`${Return[-1]}` 的设计保留，不改变其核心语义：

- `${Return[-1]}`：最近一个已完成固定步骤的运行结果
- `${Return[-2]}`：上上个已完成固定步骤的运行结果
- `${Return[0]}`：当前步骤完成后的结果（按既有约定理解）

**定义**：
`Return` 是统一运行时上下文中的**结果历史栈视图（history view）**，用于基于执行顺序的相对引用。

它解决的问题仍然是：
> 测试步骤没有稳定编号，因此需要通过相对位置引用最近的执行结果。

---

### 2.3 set / get 是统一上下文的“命名访问能力”

`set` / `get` 的定位保留，并补充定义如下：

| 关键字 | 角色 | 本质 |
|--------|------|------|
| `set` | 向运行时上下文写入命名值 | named entry write |
| `get` | 从运行时上下文读取命名值或已有结果 | named/path entry read |

**关键点**：
- `set` / `get` 不是“变量空间 vs Return 空间”的切换
- 而是在**同一个上下文**里，增加“按名字稳定存取”的能力
- `${Return[-1]}` 适合近距离引用
- `set` / `get` 适合长链路、重复对象、多实例场景下的稳定命名引用

例如：

```xml
<test_step action="send" model="OrderAPI" data="D001"/>
<test_step action="set" model="" data="first_order_no=${Return[-1].orderNo}"/>

<test_step action="send" model="OrderAPI" data="D002"/>
<test_step action="set" model="" data="second_order_no=${Return[-1].orderNo}"/>
```

此时：
- `Return[-1]` / `Return[-2]` 仍然存在
- `first_order_no` / `second_order_no` 成为统一上下文中的命名入口

---

### 2.4 自动返回值提取是模型能力，不是用户补丁步骤

在 UI 自动化及未来的结构化业务步骤中，很多步骤完成后会自然产生关键业务值。

例如：
- 创建询价单 → 生成 `inquiryNo`
- 创建订单 → 生成 `orderNo`
- 登录成功 → 获取 `currentUser` / `token`

**设计结论**：
- 这些值应优先由**模型定义的自动返回值提取规则**自动获取
- 不应默认要求用户再额外写一个 `get` 步骤补取
- `get` 只作为自动提取未覆盖到时的显式补充能力

---

### 2.5 evaluate 保留，但必须严格受限

`evaluate` 不是核心主路径关键字，而是：

> **Web 场景下的受限运行时兜底能力（escape hatch）**

**设计结论**：
- 保留 `evaluate`
- 但必须明确其为 **Web-only**
- 优先用于“读取浏览器运行时值”
- 不推荐作为常规业务测试主路径能力
- 优先级低于“自动返回值提取”与“get 显式读取”

---

## 3. 统一运行时上下文模型

### 3.0 逻辑视图 vs 物理实现

**重要说明**：从业务理解角度，用户会感知到多种“空间”概念，但从框架实现角度，所有数据都存储在一个统一的运行时上下文中。

#### 逻辑视图（用户视角）

用户在使用时会感知到不同的“空间”：

- **步骤历史空间**：我要取上一步的结果 → 使用 `${Return[-1]}`
- **命名变量空间**：我要存一个有名字的值 → 使用 `set` / `get`
- **自动提取结果空间**：关键业务值由模型自动保存到当前步骤结果中
- **结构化对象空间**（未来）：我要访问某个对象实例 → 使用路径访问

#### 物理实现（框架视角）

框架内部只维护一个统一的运行时上下文，不同“空间”只是对该上下文的不同逻辑分区：

```python
RuntimeContext = {
    "history": [...],      # Return[-1] 访问这里
    "named": {...},       # set/get 访问这里
    "objects": {...},     # 未来扩展访问这里
}
```

**核心理解**：
- `${Return[-1]}` 和 `get var_name` 本质上都是“从统一运行时上下文中读取值”
- 区别只在于访问的是不同的逻辑分区，以及语法形式不同
- 从这个角度看，`${Return[-1]}` 可以理解为 `get` 在 history 视图上的一种特殊写法

---

### 3.1 逻辑结构

统一运行时上下文在逻辑上可理解为：

```python
RuntimeContext = {
    "history": [...],   # 步骤结果历史
    "named": {...},    # 命名变量入口
    "objects": {...},  # 未来预留：结构化对象入口
}
```

### 3.2 history：结果历史

`history` 用于保存每一步关键字的执行结果。

示意：

```python
context.history = [
    {"status": 200, "orderNo": "A001"},
    "A001",
    {"status": 200, "orderNo": "A002"},
    "A002",
]
```

访问方式：
- `${Return[-1]}`
- `${Return[-2]}`
- `${Return[-1].orderNo}`

---

### 3.3 named：命名入口

`named` 用于保存通过 `set` 写入的稳定命名值。

示意：

```python
context.named = {
    "first_order_no": "A001",
    "second_order_no": "A002",
    "token": "abc123",
}
```

访问方式：
- `get first_order_no`
- 未来可扩展 `${first_order_no}`

---

### 3.4 objects：未来预留结构化路径能力

首版设计不强制落地独立“数据空间”，但为未来结构化访问预留对象入口。

示意：

```python
context.objects = {
    "Order": {
        "first": {"orderNo": "A001"},
        "second": {"orderNo": "A002"},
    }
}
```

未来如需扩展，可支持：
- `${Order.first.orderNo}`
- `get Order.first.orderNo`

**注意**：
这属于统一运行时上下文的扩展能力，不单独定义为另一套“数据空间”机制。

---

## 4. 关键字职责定义

### 4.1 所有测试步骤都必须写入 history

统一约束：
> **每个测试步骤执行完成后，必须向运行时上下文的 `history` 追加一条记录。**

这样可以保证：
- `${Return[-1]}` 的语义连续
- 任意步骤都可以作为后续步骤的数据来源
- 测试链路具备完整可追溯性

---

### 4.2 set 的语义

`set` 的职责是：
1. 从输入表达式中解析出值
2. 将值写入统一运行时上下文的 `named`
3. 将本次“实际写入的值”追加到 `history`

**推荐语义**：

```xml
<test_step action="set" model="" data="order_no=${Return[-1].orderNo}"/>
```

执行结果：
- `context.named["order_no"] = "..."`
- `context.history.append("...")`

---

### 4.3 get 的语义

`get` 的职责是：
1. 从统一运行时上下文读取命名值，或未来支持按路径读取结构化值
2. 将本次“实际读取到的值”再次追加到 `history`

**推荐语义**：

```xml
<test_step action="get" model="" data="order_no"/>
```

执行结果：
- 读取 `context.named["order_no"]`
- `context.history.append(value)`

这样 `get` 的结果也可继续被 `${Return[-1]}` 引用。

---

### 4.4 Return 的语义边界

`Return` 适合：
- 最近一步或附近几步的快速结果引用
- 不需要额外命名的短链路场景

`Return` 不适合：
- 多个同类对象并存的长期引用
- 需要高可读性、高稳定性的业务中间值保存

因此：
- **简单链路** 优先使用 `${Return[-1]}`
- **复杂链路** 推荐使用 `set` / `get` 给关键值命名

---

### 4.5 自动返回值提取的语义

自动返回值提取的职责是：
1. 在业务步骤执行成功后，根据模型定义自动读取关键业务值
2. 将读取结果组装为结构化返回值
3. 将结构化结果写入当前步骤的 `history`
4. 让后续步骤可以直接通过 `${Return[-1].字段}` 使用这些值

**核心原则**：
- 自动返回值提取是**模型能力**，不是 case 层临时补丁能力
- 用户优先写业务步骤，不应被迫额外补 `get` 步骤来获取关键业务值
- `get` 用于自动提取未覆盖到时的显式补充读取

---

## 5. 自动返回值提取设计

### 5.1 设计定位

自动返回值提取（Auto Capture）用于解决以下问题：

- 用户执行了一个完整业务步骤（如创建询价单）
- 页面已经产生了关键业务结果（如询价单号）
- 这些结果是后续步骤高度依赖的值
- 但如果要求用户再手写一个 `get` 步骤去抓取，会增加用例负担

**因此**：框架应允许在模型中定义“关键返回值提取规则”，在步骤执行成功后自动抓取并保存。

---

### 5.2 用户使用效果

#### 场景：创建询价单

用户只写：

```xml
<test_step action="type" model="InquiryCreate" data="I001"/>
```

假设 `InquiryCreate` 模型定义了自动提取规则：
- `inquiryNo`
- `status`

那么执行完成后，框架自动将当前步骤返回值写为：

```python
{
    "inquiryNo": "XJ20260407001",
    "status": "待审核"
}
```

后续步骤即可直接使用：

```xml
<field name="inquiryNo">${Return[-1].inquiryNo}</field>
```

**用户无需再额外写一个 `get` 步骤。**

---

### 5.3 使用优先级

统一建议优先级如下：

1. **自动返回值提取**（主路径，最佳体验）
2. **get**（显式补充读取）
3. **evaluate**（Web 场景受限兜底）

即：
- 能由模型自动提取的，优先自动提取
- 自动提取没有覆盖到的，再使用 `get`
- `get` 也无法自然表达的特殊浏览器运行时场景，最后才考虑 `evaluate`

---

### 5.4 自动提取定义位置

**主方案：定义在模型中**

原因：
- 返回值提取规则属于“模型能力”
- 它描述的是“这个页面/业务模型完成后，哪些值是关键结果”
- 与具体数据行无关，也不应主要放在 case 步骤里重复声明

**不推荐主放在数据表中**：
- 数据表是输入，不宜承担输出语义

**不推荐主放在 case 步骤中**：
- case 应关注业务流程，而不是重复定义模型提取规则

---

### 5.5 自动提取的返回值形态

自动提取成功后，建议当前步骤返回值直接写成结构化对象：

```python
{
    "inquiryNo": "XJ20260407001",
    "status": "待审核"
}
```

而不是仅返回布尔值或“执行过哪些字段”的过程性结果。

**原因**：
- 便于 `${Return[-1].字段}` 直接使用
- 更符合业务语义
- 与 `send` / `DB` / `run` 等结构化返回风格一致

---

### 5.6 触发时机

自动返回值提取应在以下时机触发：

- 当前业务步骤执行成功后
- 主要面向 `type` 这类 UI 模型驱动步骤
- 后续如有需要，可扩展到其他支持模型驱动结果提取的步骤

首版建议聚焦：
- **UI 场景**
- **模型定义明确的关键业务结果**

---

### 5.7 取值来源

自动提取规则未来可支持多种来源，但首版设计应明确抽象层级。

可能的取值来源包括：
- 元素文本
- 元素属性值
- 页面显示值
- 页面运行时对象值（谨慎）
- 桌面端 OCR 文本（未来）

**原则**：
- 优先使用模型可表达的标准读取方式
- 不应默认要求用户使用 `evaluate` 实现自动提取

---

### 5.8 自动提取失败的处理原则

如果一个步骤定义了自动提取规则，但提取失败：

**首版建议**：
- 默认视为当前步骤失败，或至少视为“业务结果不完整”错误
- 不允许 silent fail

原因：
- 如果框架声明“该步骤完成后会自动得到关键业务值”，但最终没有得到，后续链路很可能已经不可用
- 这种失败应尽早暴露，而不是把问题推迟到后续 `verify` 或业务步骤里爆炸

---

## 6. evaluate 关键字的定位与约束

### 6.1 evaluate 为什么可以保留

`evaluate` 有其实际价值：
- 可读取浏览器上下文中的运行时状态
- 可处理标准关键字暂时未覆盖到的浏览器内部值
- 比新增一批零散专用关键字更克制

例如：
- `document.title`
- `window.xxx`
- `localStorage.getItem('token')`

因此，`evaluate` 可以保留为**兜底能力**。

---

### 6.2 evaluate 为什么不能成为主路径能力

`evaluate` 存在天然风险：

1. **会绕开模型驱动与 DSL 约束**  
   用户直接写 JS，容易让测试逻辑回退成脚本驱动，而不是模型/数据驱动。

2. **平台不统一**  
   当前实现依赖浏览器 `page.evaluate()`，天然偏 Web，不适合作为跨平台核心能力。

3. **容易被滥用成逃生舱**  
   本应补足模型能力、标准关键字能力、自动提取能力的问题，可能被一句 JS 临时糊过去，长期侵蚀框架质量。

---

### 6.3 evaluate 的正式定位

`evaluate` 的正式定位应为：

> **Web 场景下的受限运行时兜底能力（escape hatch），不作为核心业务主路径关键字。**

---

### 6.4 evaluate 的使用优先级

用户应遵循以下优先顺序：

1. 能通过模型自动返回值提取完成 → 不用 `evaluate`
2. 能通过 `get` 显式读取完成 → 不用 `evaluate`
3. 只有浏览器运行时内部状态无法被标准 DSL 自然表达时 → 才使用 `evaluate`

---

### 6.5 evaluate 的使用约束

1. `evaluate` 仅适用于 **Web 浏览器驱动**。
2. `evaluate` 优先用于**读取值**，不鼓励作为“改页面状态 / 注入业务逻辑”的常规手段。
3. 不应在常规业务用例主链路中大量使用 `evaluate`。
4. `evaluate` 返回值必须按统一运行时上下文规则写入 `history`。
5. `dict/list` 结果必须保留结构，不允许无条件 `str()`。

---

## 7. 返回值存储规范

### 7.1 存储类型原则

| 返回值类型 | 存入 history 的值 |
|-----------|------------------|
| dict / list | 原样存入 |
| str | 原样存入 |
| bool | 原样存入 |
| None | 原样存入 |
| 其他基础类型 | 原样存入或按现有安全方式转换 |

**禁止**：
- 不得对 dict/list 使用 Python `str()` 转换后再写入 history
- 不得将结构化结果无条件降级为字符串

---

### 7.2 evaluate 的修复要求

当前 `evaluate` 存在的问题是：
```python
result_str = str(result) if result is not None else ""
self.store_return(result_str)
```

这会导致：
- dict/list 被转成 Python repr 字符串
- 后续字段访问能力丢失
- 与 `send` / `run` / `DB` 的结构化返回不一致

**修复要求**：
- `dict/list`：原样写入 history
- `None`：写入 `None`
- 基础值：保持原始类型或按必要规则处理

---

## 8. 当前关键字的统一写入要求

以下关键字执行完成后，均应写入统一运行时上下文的 `history`：

| 关键字 | 建议写入值 |
|--------|-----------|
| `navigate` | `True` |
| `launch` | `True` |
| `type` | 自动提取后的结构化业务结果，或实际处理后的输入结果 |
| `send` | 响应结果 dict |
| `verify` | 验证结果 list |
| `evaluate` | JS 执行结果（保持结构） |
| `run` | JSON 结果 / stdout / `None` |
| `DB` | SQL 结果 / `None` |
| `assert` | `True` |
| `set` | 实际写入值 |
| `get` | 实际读取值 |
| `wait` | `True` |
| `close` | `True` |
| `clear` | `True` |
| `upload_file` | `True` |

说明：
- 某些关键字失败时可写入 `None`，或按现有异常语义处理
- 关键目标不是改变业务逻辑，而是保证 history 链的连续性与一致性
- `type` 是自动返回值提取的重点承载关键字

---

## 9. 为什么不单独定义“数据空间”

本次设计明确不将“数据空间”定义为独立于 Return / set / get 的第四套机制，原因如下：

1. **概念会重复**  
   如果再引入独立数据空间，会与 Return、命名变量形成多套并行心智模型，增加理解成本。

2. **多实例场景会快速复杂化**  
   例如同一流程中创建两次订单，如果仅有 `Order.orderNo` 这样的路径，将无法明确区分取的是哪个订单号。

3. **问题本质不是要多一套空间，而是要统一运行时上下文**  
   `Return` 解决相对步骤引用问题，`set/get` 解决命名引用问题，自动返回值提取解决“默认关键结果获取”问题，这三者统一后已能覆盖大多数场景。

**结论**：
- 当前阶段采用“一个统一运行时上下文”的方案
- 如未来确实需要结构化对象访问，也应作为该上下文内部的路径能力扩展，而不是另起一套机制

---

## 10. 核心设计约束（草案）

以下约束为本设计落地时必须遵守的规则，后续实现完成后再同步更新到正式核心项目文档。

### 10.1 统一上下文约束

1. RodSki 在单个 case 执行期间，只维护**一个统一运行时上下文**。
2. `Return`、`set`、`get`、自动返回值提取，都属于该上下文的访问机制，不允许再引入与之并行的第二套运行时存储语义。
3. 运行时上下文仅对**当前 case** 生效，不跨 case 共享。

### 10.2 Return 语义约束

1. `${Return[-N]}` 的核心语义保持不变，仍表示对最近固定步骤结果的相对引用。
2. `Return` 是统一运行时上下文的历史视图，不应再被设计为独立变量池。
3. 动态步骤与固定步骤的 Return 关系，继续遵循现有核心设计约束，不在本设计中擅自改变。

### 10.3 set / get 约束

1. `set` / `get` 属于运行时上下文的命名访问能力。
2. `set` 的职责是把一个值写入统一上下文中的命名入口。
3. `get` 的职责是从统一上下文中按名称或路径读取值。
4. `set` / `get` 执行完成后，也必须写入 history，保证步骤链连续。

### 10.4 自动返回值提取约束

1. 自动返回值提取是**模型能力**，不应主要依赖 case 层重复定义。
2. 关键业务步骤执行成功后，若模型定义了自动提取规则，应自动将提取结果写入当前步骤返回值。
3. 自动提取优先级高于显式 `get`，应作为主路径能力。
4. 自动提取失败不得 silent fail，应作为可见错误暴露。

### 10.5 evaluate 约束

1. `evaluate` 仅作为 **Web-only 兜底能力** 使用。
2. `evaluate` 不属于核心业务主路径关键字。
3. 优先通过自动返回值提取与 `get` 解决取值问题，不得把 `evaluate` 作为默认取值手段。
4. `evaluate` 的结构化返回值必须保留结构。

### 10.6 返回值写入约束

1. 每个测试步骤执行完成后，都必须向统一运行时上下文的 `history` 追加一条记录。
2. 结构化返回值（dict/list）必须保留结构，不得无条件转成字符串。
3. 不允许因为关键字类型不同而出现“有的步骤入 history，有的步骤不入 history”的不一致行为。

### 10.7 兼容性约束

1. 不改变 `${Return[-N]}` 的基本写法。
2. 不在当前阶段新增新的独立关键字体系。
3. 不要求用户在首版中学习新的多空间概念，用户只需要理解“统一运行时上下文”即可。

## 11. 用户使用说明（草案）

以下内容为未来面向用户文档的说明草稿，当前先保存在设计文档中。

### 11.1 用户需要理解的核心概念

执行一个 case 时，RodSki 会维护一份统一运行时上下文。

用户在用例中拿值，优先有三种方式：

1. **模型自动返回值提取**：关键业务值自动进入当前步骤返回值
2. **按最近步骤结果取值**：使用 `${Return[-1]}`、`${Return[-2]}`
3. **按名字取值**：先 `set` 保存，再用 `get` 或后续引用读取

---

### 11.2 推荐使用顺序

建议用户按以下顺序思考：

1. 这个值是否应该由模型自动返回？
2. 如果不是，是否可以直接用 `${Return[-1]}`？
3. 如果还不够清晰，是否应该用 `set` 命名？
4. 如果标准方式都不适用，才考虑 `get` 补充读取或 `evaluate` 兜底

---

### 11.3 什么时候用自动返回值提取

适合场景：
- 业务步骤天然会产生关键结果
- 后续步骤高度依赖这些结果
- 希望减少样板式 `get` 步骤

示例：

```xml
<test_step action="type" model="InquiryCreate" data="I001"/>
```

如果模型已定义自动提取 `inquiryNo`，后续即可直接使用：

```xml
<field name="inquiryNo">${Return[-1].inquiryNo}</field>
```

---

### 11.4 什么时候用 Return

适合场景：
- 只需要引用最近一步或附近几步的结果
- 值只会短暂使用一次
- 用例链路较短，步骤关系清晰

示例：

```xml
<test_step action="send" model="LoginAPI" data="D001"/>
<test_step action="verify" model="LoginAPI" data="V001"/>
```

如果后续步骤只需要上一步结果，可继续使用 `${Return[-1]}`。

---

### 11.5 什么时候用 set / get

适合场景：
- 某个值要跨多个步骤反复使用
- 同一流程里有多个相似对象（如两个订单号、两个 token）
- 希望让用例语义更清晰，而不是依赖 `${Return[-2]}`、`${Return[-3]}` 这类相对位置引用

示例：

```xml
<test_step action="type" model="InquiryCreate" data="I001"/>
<test_step action="set" model="" data="first_inquiry_no=${Return[-1].inquiryNo}"/>

<test_step action="type" model="InquiryCreate" data="I002"/>
<test_step action="set" model="" data="second_inquiry_no=${Return[-1].inquiryNo}"/>
```

这种写法比一直依赖 `${Return[-1]}` / `${Return[-2]}` 更清晰。

---

### 11.6 什么时候用 evaluate

适合场景：
- 只有浏览器运行时内部状态才能拿到的值
- 标准模型能力、自动提取、`get` 都无法覆盖

不适合场景：
- 常规业务主流程取值
- 大量替代模型驱动能力
- 用 JS 改页面状态作为常规测试手段

---

### 11.7 使用建议

1. **优先让模型自动返回关键值**  
   如果某个业务步骤完成后天然会产生关键结果，应优先通过模型定义自动提取，而不是强迫用户补 `get`。

2. **简单链路优先用 Return**  
   如果值只在最近一步附近使用，优先用 `${Return[-1]}`，写法更直接。

3. **复杂链路优先给值命名**  
   如果值很重要、会跨多个步骤使用、或者同类值会出现多次，优先通过 `set` 保存命名值。

4. **避免过度依赖深层 Return 索引**  
   `${Return[-4]}`、`${Return[-5]}` 这类写法虽可用，但可读性较差，建议改为 `set` 命名。

5. **把 evaluate 当作兜底，不当作主路径**  
   `evaluate` 可用，但不应成为默认取值方式。

---

### 11.8 当前阶段的用户心智模型

用户当前只需要记住一句话：

> RodSki 在执行过程中只有一个统一运行时上下文；优先由模型自动返回关键值，`${Return[-1]}` 用于按步骤历史取值，`set/get` 用于按名字取值，`evaluate` 只作为 Web 场景下的受限兜底能力。

## 13. 实例化需求、场景法与验收用例

本节用具体业务场景说明本设计要解决的真实需求，并给出可验收的用例标准。

### 13.1 实例化需求

#### 需求 1：创建单据后自动获得关键编号

**业务背景**：
在 UI 自动化场景中，很多核心业务动作执行完成后，会生成后续步骤必须依赖的关键编号，例如：
- 询价单号
- 订单号
- 调货单号
- 退货单号

**现状问题**：
用户往往需要这样写：

```xml
<test_step action="type" model="InquiryCreate" data="I001"/>
<test_step action="get" model="" data="css=.inquiry-no"/>
<test_step action="set" model="" data="inquiry_no=${Return[-1]}"/>
```

这会导致：
- 用例啰嗦
- `get` 步骤只是框架补洞，不是业务动作
- 用户已经表达了“创建询价单”，却还要自己再补抓关键结果

**目标效果**：
用户只写：

```xml
<test_step action="type" model="InquiryCreate" data="I001"/>
```

框架自动将：
- `inquiryNo`
- `status`

写入当前步骤返回值，后续直接通过 `${Return[-1].inquiryNo}` 使用。

---

#### 需求 2：同一流程中保存多个同类业务值

**业务背景**：
同一流程中可能连续创建两次单据，例如：
- 创建两张询价单
- 创建两个订单
- 获取两个 token

**现状问题**：
如果只依赖 `${Return[-1]}` / `${Return[-2]}`，随着步骤增多，用户很难稳定表达“我要第一个单号”还是“第二个单号”。

**目标效果**：
用户可以通过：

```xml
<test_step action="type" model="InquiryCreate" data="I001"/>
<test_step action="set" model="" data="first_inquiry_no=${Return[-1].inquiryNo}"/>

<test_step action="type" model="InquiryCreate" data="I002"/>
<test_step action="set" model="" data="second_inquiry_no=${Return[-1].inquiryNo}"/>
```

把两个值稳定命名保存到统一运行时上下文中。

---

#### 需求 3：自动提取覆盖不到时，显式补充读取

**业务背景**：
有些页面值不是核心默认返回值，或者模型暂时没有配置自动提取规则。

**目标效果**：
用户仍可通过 `get` 补充读取：

```xml
<test_step action="get" model="" data="special_flag"/>
```

或未来扩展的路径方式读取统一上下文中的值。

---

#### 需求 4：保留 evaluate 作为受限兜底能力

**业务背景**：
某些浏览器运行时值无法通过标准模型读取自然表达，例如：
- `document.title`
- `window.__APP_STATE__`
- `localStorage` 中的值

**目标效果**：
允许用户在标准方式都不适用时使用 `evaluate`，但必须在文档和约束中明确：
- 它是 Web-only
- 它是兜底能力，不是主路径
- 它优先用于读取，而不是大范围改状态

---

#### 需求 5：执行日志既要适合人看，也要适合 AI Agent 判断

**业务背景**：
测试用例执行后，不仅人类需要快速判断步骤是否正确、关键值是否成功获取，AI Agent 也需要基于执行输出自动检查：
- 用例是否写得合理
- 自动返回值提取是否生效
- `get` / `set` 是否使用正确
- `evaluate` 是否被滥用
- 当前失败属于动作失败、自动提取失败，还是运行时取值失败

**现状问题**：
如果只有自然语言日志：
- 人可以大致看懂，但日志级别不统一时很难稳定回溯
- AI Agent 只能依赖解析文本，容易脆弱
- Info 模式和 Debug 模式如果输出差异过大，会影响后续自动判断

**目标效果**：
- 提供 **Info 模式**：给用户快速阅读，显示步骤摘要、关键返回值、变量写入和错误摘要
- 提供 **Debug 模式**：给排错与深度分析，显示参数解析、自动提取过程、history/named 变化、失败根因
- 无论 Info / Debug，都必须提供**独立于日志的结构化执行结果**，供 AI Agent 稳定消费

---

**预期效果**：
- 当前步骤应报错或明确标记失败
- 不允许 silent fail
- 日志中应明确指出是“自动返回值提取失败”而不是普通输入失败

#### 场景 D：特殊浏览器运行时值通过 evaluate 读取

**约束说明**：

**预期效果**：

**预期效果**：
- AI Agent 不依赖自然语言日志也能判断问题
- AI Agent 可识别每一步的返回值来源（如 keyword_result / auto_capture / get_named / evaluate）
- AI Agent 可判断：自动提取是否生效、用例是否过度依赖深层 Return、evaluate 是否被滥用

### 13.3 验收用例

以下验收用例用于判断本设计是否真正落地。

#### 验收用例 1：自动返回值提取成功

**步骤**：
1. 在 `rodski-demo` 示例网站上执行 `type DemoForm D001`
2. 对应模型已配置自动提取 `resultId`
3. 检查当前步骤返回值

**验收标准**：
- `history` 中该步骤返回值为结构化对象
- 返回值包含 `resultId`
- 后续 `${Return[-1].resultId}` 可正常解析

---

#### 验收用例 2：无需额外 get 步骤即可复用关键值

**步骤**：
1. 在 `rodski-demo` 示例网站上执行表单提交步骤
2. 下一步直接在验证数据表中使用 `${Return[-1].resultId}`

**验收标准**：
- 用例可执行成功
- 无需额外补 `get`
- 验证步骤正确读取到自动提取的关键值

---

#### 验收用例 3：set 保存多个同类值

**步骤**：
1. 在 `rodski-demo` 示例网站上连续提交两次记录
2. 每次提交后分别用 `set` 保存 `first_result_id`、`second_result_id`
3. 后续通过 `get` 或引用读取

**验收标准**：
- 两个变量值彼此独立
- 命名值可重复读取
- 不依赖深层 Return 索引也能保持语义稳定

---

#### 验收用例 4：get 读取统一运行时上下文值

**步骤**：
1. 先通过 `set` 写入命名变量
2. 执行 `get`
3. 检查返回值是否再次进入 history

**验收标准**：
- `get` 能读到命名值
- `get` 结果会再次写入 history
- 后续 `${Return[-1]}` 指向 `get` 读取结果

---

#### 验收用例 5：evaluate 保留结构化返回能力

**步骤**：
1. 在 `rodski-demo` 示例网站页面执行 `evaluate`
2. 返回一个 JS 对象或数组
3. 检查 history 存储结果

**验收标准**：
- dict/list 不被 `str()` 降级
- 可通过 `${Return[-1].field}` 继续访问
- 与 `send` / `run` / `DB` 的结构化返回行为一致

---

#### 验收用例 6：自动提取失败时可见报错

**步骤**：
1. `rodski-demo` 中某模型定义了必须自动提取的关键字段
2. 示例页面上实际未出现该字段
3. 执行对应业务步骤

**验收标准**：
- 当前步骤失败
- 日志能明确区分“业务动作成功但自动提取失败”或统一报为步骤失败
- 不允许 silent fail

---

#### 验收用例 7：evaluate 不替代主路径能力

**步骤**：
1. 对一个本可通过 `rodski-demo` 模型自动提取的字段，设计一个实现方案评审
2. 检查推荐使用方式

**验收标准**：
- 文档与实现都应优先推荐自动提取或 `get`
- `evaluate` 仅作为兜底能力出现
- 不鼓励在主业务流程中大量使用 `evaluate`

---

## 14. 实施优先级

| 优先级 | 内容 |
|--------|------|
| P0 | 将设计统一为“运行时上下文 + 自动返回值提取”模型，明确 Return / set / get / evaluate 的关系 |
| P1 | 补全所有关键字的 history 写入，保证步骤链连续 |
| P2 | 修复 `evaluate` 的结构化结果丢失问题，并补充使用约束 |
| P3 | 重构 `get` 关键字实现，从 UI 文本获取升级为统一运行时上下文访问 |
| P4 | 为 `type` 增加模型驱动的自动返回值提取能力 |
| P5 | 根据需要扩展统一上下文的路径访问能力 |
