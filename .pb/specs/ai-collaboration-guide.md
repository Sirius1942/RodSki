## 概述

### 为什么需要 AI 协作？

传统测试用例编写面临的挑战：
- **重复劳动**: 大量相似的模型定义和数据表
- **学习成本**: 需要记忆框架语法和规范
- **维护困难**: 用例数量增加后难以管理
- **质量不稳定**: 人工编写容易出错

### AI 协作的优势

- ✅ **快速生成**: 从需求到用例，几分钟完成
- ✅ **规范一致**: 自动遵循框架约束和命名规范
- ✅ **智能补全**: 根据上下文推断模型和数据
- ✅ **质量保证**: 自动检查语法和逻辑错误
- ✅ **持续优化**: 根据反馈不断改进

### 协作原则

1. **人类负责**: 需求定义、业务逻辑、验收标准
2. **AI 负责**: 代码生成、格式规范、重复工作
3. **共同负责**: 质量检查、问题修复、持续改进

## AI 协作工作流

### 标准流程 (5步)

```
需求输入 → AI生成 → 人工审查 → 执行验证 → 迭代优化
```

#### Step 1: 需求输入

**人类提供**:
- 测试场景描述
- 业务流程说明
- 验证点要求
- 特殊约束条件

**示例**:
```
我需要测试用户登录功能：
1. 打开登录页面 http://localhost:8000
2. 输入用户名 admin 和密码 123456
3. 点击登录按钮
4. 验证跳转到首页，显示"欢迎, admin"
```

#### Step 2: AI 生成

**AI 输出**:
- Case XML (测试用例)
- model.xml (模型定义)
- data/*.xml (测试数据)
- 说明文档

#### Step 3: 人工审查

**检查项**:
- [ ] 业务逻辑是否正确
- [ ] 定位器是否准确
- [ ] 数据是否完整
- [ ] 命名是否规范

#### Step 4: 执行验证

**运行测试**:
```bash
python3 rodski/ski_run.py rodski-demo/DEMO/demo_full/case/demo_case.xml
```

**检查结果**:
- 测试是否通过
- 日志是否正常
- 截图是否正确

#### Step 5: 迭代优化

**反馈给 AI**:
- 报告问题
- 提出改进建议
- 补充遗漏场景

```

- [验证点1]
- [验证点2]

5. 点击注册按钮
6. 验证跳转到欢迎页面，显示"注册成功"

验证点:
- 页面URL包含 /welcome

## 最佳实践

### 1. 清晰的需求描述

**❌ 不好的描述**:
```
帮我写个登录测试
```

**✅ 好的描述**:
```
我需要测试 Web 登录功能：
- 页面: http://localhost:8000/login
- 输入: 用户名(admin) + 密码(123456)
- 操作: 点击登录按钮
- 验证: 跳转到首页，显示"欢迎, admin"
- 清理: 关闭浏览器
```

### 2. 提供页面结构信息

**方式1: 提供 HTML 片段**
```html
<form id="loginForm">
  <input id="username" name="username" />
  <input id="password" name="password" type="password" />
  <button id="loginBtn">登录</button>
</form>
```

**方式2: 提供元素定位信息**
```
- 用户名输入框: id="username"
- 密码输入框: id="password"
- 登录按钮: id="loginBtn"
```

**方式3: 提供截图**
```
我上传了页面截图，请根据截图生成测试用例。
```

### 3. 明确验证点

**❌ 模糊的验证**:
```
验证登录成功
```

**✅ 明确的验证**:
```
验证点:
1. URL 变为 http://localhost:8000/dashboard
2. 页面标题显示"控制台"
3. 右上角显示用户名"admin"
4. 左侧菜单显示5个菜单项
```

### 4. 分步骤迭代

**不要一次性要求太多**:
```
❌ 帮我生成整个系统的所有测试用例
```

**分阶段进行**:
```
✅ 第一步: 先生成登录功能的测试用例
✅ 第二步: 基于登录用例，添加用户管理测试
✅ 第三步: 添加订单管理测试
```

### 5. 及时反馈和修正

**发现问题立即反馈**:
```
生成的用例有个问题：
- 定位器 id="submitBtn" 应该是 id="loginBtn"
- 验证数据中缺少了 welcomeMsg 字段
请修正这两处。
```

### 6. 复用和扩展

**复用现有资源**:
```
我需要添加一个新的登录测试用例，使用不同的测试数据。
请复用现有的 LoginForm 模型，只生成新的数据行 L002。
```

## 质量检查清单

### AI 生成后的检查项

#### ✅ Case XML 检查

- [ ] `execute="是"` 正确设置
- [ ] `id` 唯一且有意义
- [ ] `title` 清晰描述用例目的
- [ ] `component_type` 正确分类
- [ ] `expect_fail` 在负面用例中正确标记
- [ ] `pre_process` / `test_case` / `post_process` 结构合理
- [ ] 每个 `test_step` 的 `action` 是有效关键字
- [ ] `model` 和 `data` 引用正确

#### ✅ model.xml 检查

- [ ] `model name` 与用例中引用一致
- [ ] `element name` 与数据表字段一致
- [ ] 定位器类型正确 (id/css/xpath/name/vision)
- [ ] 定位器值准确
- [ ] 接口模型包含 `_method` 和 `_url`
- [ ] `auto_capture` 配置正确（如果使用）

#### ✅ 数据表 XML 检查

- [ ] `datatable name` 与文件名一致
- [ ] `row id` 唯一
- [ ] `field name` 与模型元素一致
- [ ] 数据值符合业务规则
- [ ] 验证表 (`_verify.xml`) 包含期望值

#### ✅ 逻辑检查

- [ ] 测试步骤顺序合理
- [ ] 数据依赖关系正确
- [ ] 验证点覆盖完整
- [ ] 错误处理适当
- [ ] 清理操作完整

## 注意事项

### ⚠️ AI 的局限性

1. **不了解实际页面**: AI 无法看到真实页面，需要你提供准确信息
2. **可能产生幻觉**: 生成的定位器可能不存在，需要验证
3. **缺乏业务理解**: 验证点可能不完整，需要补充
4. **格式可能有误**: 需要人工检查 XML 语法

### ✅ 如何规避

1. **提供详细信息**: 越详细越好
2. **分步验证**: 生成后立即测试
3. **及时反馈**: 发现问题立即修正
4. **建立规范**: 形成团队的提示词标准
