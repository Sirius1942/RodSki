## 1. 核心定位转变

### 1.1 原定位
- 关键字驱动的自动化测试框架
- 面向测试工程师
- 用于编写和执行测试用例

### 1.2 新定位
**LLM 时代的 AI Agent 操作系统 + 活文档系统**

**核心能力**：
- 关键字驱动（Keyword-Driven）
- 数据驱动（Data-Driven）

**核心目标**：
- 协助 AI Agent 快速操作电脑（Web/PC/移动端）
- 结构化记录操作方法（活文档）
- 解决 AI use computer 时每步都需要模型判断导致的性能问题

---

## 2. 目标用户

### 2.1 主要用户
**AI Agent 工具**（如 OpenClaw、Claude Code）

RodSki 本身**不开发 Agent 能力**，而是作为 Agent 的**操作执行层**和**知识记录层**。

### 2.2 次要用户
- 开发 Agent 集成的工程师
- 维护操作知识库的工程师

## 3. Agent 与 RodSki 的交互模式

### 3.1 模式 1：探索模式（Exploration Mode）

**场景**：Agent 首次接触某个网站或应用

**流程**：
1. Agent 使用自己的能力探索网站/应用（如探索网页邮箱）
2. 探索过程中，Agent 识别关键操作路径
3. 使用 RodSki 的 XML 格式记录：
   - **模型**（Model）：页面元素定位信息
   - **操作方法**：如何操作网页邮箱发邮件
   - **数据结构**：需要哪些输入数据

**输出**：可执行的 XML 用例 + 模型 + 数据模板

### 3.2 模式 2：执行模式（Execution Mode）

**场景**：Agent 需要执行已探索过的功能

**流程**：
1. Agent 判断使用哪些已有用例
2. 根据当前任务需求，动态调整：
   - 修改数据表（Data XML）
   - 插入/删除测试步骤（Case XML）
3. 调用 RodSki 执行
4. 获取执行结果

**核心价值**：
- **减少 Token 消耗**：不需要重新探索页面结构
- **减少执行时间**：不需要每步都用模型判断
- **提高可靠性**：使用已验证的操作路径

## 4. 活文档定义

### 4.1 什么是活文档

**不是**：简单的文本描述或静态 XML 记录
**而是**：真正可执行的操作知识

### 4.2 活文档的特征

1. **可执行性**：XML 文件可以直接被 RodSki 执行
2. **可复用性**：调整数据即可用于不同场景
3. **可组合性**：多个操作可以组合成复杂流程
4. **可演化性**：Agent 可以基于执行结果优化操作路径

### 4.3 活文档的结构

```
product/<domain>/<site>/
├── case/      # 操作流程（可执行的步骤序列）
├── model/     # 页面/接口模型（元素定位信息）
├── data/      # 数据模板（可替换的输入数据）
├── fun/       # 自定义函数（扩展能力）
└── result/    # 执行结果（反馈给 Agent）
```

## 5. 核心价值主张

### 5.1 解决的问题

**AI use computer 的性能瓶颈**：
- 每个操作都需要模型判断 → 慢、贵
- 每次都需要重新探索 → 不可靠
- 操作知识无法积累 → 重复劳动

### 5.2 RodSki 的解决方案

**探索一次，复用多次**：

| 传统 AI use computer | RodSki + Agent |
|---------------------|----------------|
| 每次都用模型探索页面 | 探索一次，记录为 XML |
| 每步都需要模型判断 | 直接执行已知步骤 |
| 操作知识无法积累 | 活文档持续积累 |
| Token 消耗大 | 仅探索时消耗 Token |
| 执行慢 | 执行快 |

## 6. 技术架构设计

### 6.1 Agent 调用方式

**方式 1：XML 生成模式**
- Agent 生成/修改 XML 文件
- 调用 RodSki CLI 执行
- 解析 result/*.xml 获取结果

**方式 2：API 调用模式**（待设计）
- Agent 通过 Python API 调用
- 动态构建操作序列
- 同步获取执行结果

### 6.2 关键字作为操作原语

当前 14 个关键字 = Agent 可用的**操作原语**（Primitive Operations）：

```
navigate  - 导航到 URL
type      - 批量输入/操作
verify    - 验证结果
send      - 发送 API 请求
wait      - 等待条件
execute   - 执行自定义函数
switch    - 切换上下文
alert     - 处理弹窗
select    - 选择下拉框
upload    - 上传文件
screenshot- 截图
scroll    - 滚动
refresh   - 刷新
close     - 关闭
```

### 6.3 动态步骤插入能力

Agent 需要能够：
1. 读取已有用例（Case XML）
2. 在 `pre_process`/`test_case`/`post_process` 中插入新步骤
3. 修改数据表（Data XML）
4. 执行并获取反馈

## 7. 关键设计决策（已明确）

### 7.1 Agent 集成接口设计 ✅

**决策**：XML + CLI 模式为核心

**实现方式**：
- RodSki 作为基础包部署到 Agent 所在的 PC 上（如 OpenClaw）
- Agent 通过 **skill** 调用 RodSki CLI
- 所有与 Agent 的交互都发生在 CLI 中

**架构**：
```
Agent (OpenClaw/Claude Code)
    ↓ (调用 skill)
RodSki CLI
    ↓ (读取)
XML 文件 (case/model/data)
    ↓ (执行)
操作原语 (14 个关键字)
    ↓ (输出)
result/*.xml (返回给 Agent)
```

---

### 7.2 活文档的版本管理 ✅

**决策**：每个 Agent 对应一套 RodSki

**当前阶段**：
- 单 Agent 单 RodSki 实例
- 不实现多执行器模式
- 未来可能扩展为多执行器

**版本管理**：
- 由 Agent 自行管理其 XML 文件
- 使用 Git 或其他版本控制工具

---

### 7.3 操作原语的扩展 ✅

**决策**：关键字暂时不扩展

**设计原则**：
- 关键字要精炼
- 保持 14 个操作原语的稳定性
- 复杂逻辑由 Agent 通过组合实现

**当前 14 个关键字**：
```
navigate, type, verify, send, wait,
execute, switch, alert, select, upload,
screenshot, scroll, refresh, close
```

---

### 7.4 错误处理和重试机制 ✅

**决策**：错误处理机制要详细，所有执行都是动态的

**核心原则**：
- 异常返回后，Agent 给出下一步命令行如何执行的判断
- 所有与 Agent 执行时的交互都发生在 CLI 中
- 不做自动重试，由 Agent 决策

**需要设计**：
1. 结构化的错误信息格式
2. 失败步骤的精确定位
3. 上下文信息的完整返回
4. 可恢复性状态的保存

## 9. 当前设计的不足分析

### 9.1 CLI 交互能力不足

**问题**：
- 当前 CLI 主要面向人类用户，不是为 Agent 设计的
- 缺少结构化的输入/输出格式
- 错误信息不够详细和结构化

**需要改进**：
1. **结构化输出**：所有输出应为 JSON 格式（可选）
2. **详细错误信息**：包含失败步骤、上下文、可恢复性状态
3. **进度反馈**：实时返回执行进度（Agent 可监控）
4. **退出码规范**：明确的退出码定义（成功/失败/部分成功）

---

### 9.2 错误处理机制不完善

**问题**：
- 错误信息格式不统一
- 缺少失败步骤的精确定位
- 缺少上下文信息（失败时的页面状态、变量值）
- 缺少可恢复性判断（能否从失败点继续）

**需要设计**：
```json
{
  "status": "failed",
  "exit_code": 1,
  "failed_step": {
    "index": 5,
    "action": "type",
    "model": "loginForm",
    "data": "validUser"
  },
  "error": {
    "type": "ElementNotFound",
    "message": "无法定位元素：#username",
    "details": "页面可能未加载完成或元素已变更"
  },
  "context": {
    "current_url": "https://example.com/login",
    "screenshot": "result/error_step5.png",
    "variables": {"Return": [...]}
  },
  "recoverable": true,
  "suggestions": [
    "增加 wait 步骤等待页面加载",
    "检查模型定义中的元素定位器"
  ]
}
```

---

### 9.3 活文档的元数据缺失

**问题**：
- XML 文件缺少元数据（创建时间、创建者、成功率）
- 无法判断 XML 的质量和可靠性
- 无法追踪 XML 的演化历史

**需要添加**：
```xml
<case>
  <metadata>
    <created_by>agent-openclaw-001</created_by>
    <created_at>2026-03-25T01:49:27Z</created_at>
    <last_validated>2026-03-25T01:49:27Z</last_validated>
    <success_rate>0.95</success_rate>
    <execution_count>20</execution_count>
    <version>1.2</version>
  </metadata>
  <!-- 原有内容 -->
</case>
```

---

### 9.4 动态步骤插入能力不足

**问题**：
- Agent 需要修改 XML 文件才能插入步骤
- 缺少运行时动态插入步骤的能力
- 缺少条件执行和循环能力

**可能方案**：
1. **CLI 参数支持**：`--insert-step "index=3,action=wait,data=5"`
2. **模板变量**：XML 中使用占位符，CLI 传入参数
3. **脚本模式**：支持简单的条件和循环逻辑

---

### 9.5 结果反馈不够丰富

**问题**：
- result/*.xml 主要记录成功/失败
- 缺少执行过程的详细信息
- 缺少性能数据（执行时间、资源消耗）
- 缺少中间结果（每步的输出）

**需要增强**：
```xml
<result>
  <summary>
    <status>success</status>
    <duration>12.5s</duration>
    <steps_executed>10</steps_executed>
  </summary>
  <steps>
    <step index="1" action="navigate" duration="2.3s" status="success">
      <output>页面加载完成</output>
    </step>
    <!-- 每步的详细信息 -->
  </steps>
  <variables>
    <Return>
      <item index="0">{"status": 200, "body": {...}}</item>
    </Return>
  </variables>
</result>
```

---

### 9.6 Skill 集成规范缺失

**问题**：
- 缺少 Agent 调用 RodSki 的 skill 定义
- 缺少 skill 的输入/输出规范
- 缺少 skill 的使用示例

**需要定义**：
- Skill 名称和参数
- 输入格式（如何传递 XML 路径、参数）
- 输出格式（如何解析结果）
- 错误处理（如何处理异常）

---

### 9.7 文档体系不匹配新定位

**问题**：
- 当前文档面向测试工程师，不是 Agent
- 缺少 Agent 集成指南
- 缺少 skill 使用文档
- 缺少错误处理最佳实践

**需要重构**：
1. Agent 集成指南（如何调用 RodSki）
2. Skill 参考文档（skill 定义和示例）
3. 错误处理指南（如何解析错误并决策）
4. 活文档规范（XML 格式和元数据）
