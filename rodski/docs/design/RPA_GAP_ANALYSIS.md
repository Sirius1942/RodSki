# RodSki 与主流 RPA 系统差距分析

**版本**: v1.0
**日期**: 2026-03-25

## 1. 当前 RodSki 能力总结

### 1.1 已支持功能
- ✅ Web 自动化 (Playwright)
- ✅ 移动端自动化 (Appium - Android/iOS)
- ✅ Windows 桌面自动化 (PyWinAuto - 基础实现)
- ✅ RESTful API 测试
- ✅ 关键字驱动 + 数据驱动
- ✅ Excel 用例管理
- ✅ 并行执行
- ✅ 数据库操作 (DB 关键字)
- ✅ Python 脚本执行 (run 关键字)

### 1.2 当前桌面自动化限制
- PyWinAuto 驱动实现非常基础（仅 73 行代码）
- 仅支持 Windows 平台
- 定位方式单一（仅 title 定位）
- 缺少键盘快捷键、剪贴板等高级操作
- **完全不支持 macOS 桌面自动化**

## 3. 关键缺失功能详解

### 3.1 🔴 高优先级缺失

#### A. macOS 桌面自动化
**现状**: 完全不支持
**影响**: 无法在 macOS 上进行桌面应用自动化
**主流方案**:
- AppleScript / JXA (JavaScript for Automation)
- Accessibility API (pyobjc)
- pyautogui (跨平台但功能有限)

#### B. 图像识别定位
**现状**: 完全不支持
**影响**: 无法处理无法通过 UI 树定位的元素
**主流方案**:
- OpenCV 模板匹配
- 深度学习目标检测 (YOLO, etc.)
- **LLM 视觉模型** (GPT-4V, Claude Vision)

#### C. 视觉 AI 定位 (LLM-based)
**现状**: 未实现
**影响**: 无法利用 AI 智能识别和定位元素
**价值**: 这是 RPA 的未来方向，可大幅降低维护成本

### 3.2 🟡 中优先级缺失

#### D. 剪贴板操作
**现状**: 不支持
**影响**: 无法实现复制粘贴自动化
**实现**: pyperclip / pyclip

#### E. OCR 文字识别
**现状**: 不支持
**影响**: 无法从图像中提取文字
**方案**: Tesseract / PaddleOCR / 云 OCR API

#### F. Office 深度集成
**现状**: 仅支持 Excel 读写
**缺失**: Word / PowerPoint / Outlook 自动化
**方案**: python-docx / python-pptx / win32com

## 5. 竞争优势与差异化

### 5.1 RodSki 的独特优势
1. **轻量级**: 无需重型 IDE，Excel + XML 即可
2. **开源**: 完全可定制，无许可证成本
3. **Python 生态**: 易于扩展和集成
4. **关键字驱动**: 学习曲线平缓
5. **多平台**: Web + Mobile + Desktop 统一框架

### 5.2 建议的差异化方向
1. **AI-First**: 深度集成 LLM 视觉定位，降低维护成本
2. **开发者友好**: 提供 Python API，支持代码 + 关键字混合
3. **云原生**: 支持容器化部署和分布式执行
4. **开源社区**: 建立插件生态

---

## 6. 总结

### 6.1 核心差距
- **桌面自动化能力不足** (尤其 macOS)
- **缺少图像识别和 OCR**
- **缺少 LLM 视觉 AI 能力**

### 6.2 建议优先级
1. 🔴 **P0**: macOS 桌面自动化 + 图像识别
2. 🟡 **P1**: LLM 视觉定位接口设计
3. 🟡 **P1**: 剪贴板 + 键盘增强
4. 🟢 **P2**: OCR + Office 深度集成

### 6.3 战略建议
- **短期**: 补齐桌面自动化基础能力，达到可用水平
- **中期**: 实现 LLM 视觉定位，建立技术护城河
- **长期**: 构建 AI-First RPA 平台，形成差异化竞争力
